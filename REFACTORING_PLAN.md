## ЧАСТЬ 1.6. ЭТАП 6 — Сохранение форматирования ответов (✅ ВЫПОЛНЕН)

### 6.1. BUG: «сплошной текст без абзацев и с нарушенной нумерацией»

**Симптом:** ответ бота в Telegram приходит единым блоком текста — нет абзацев
(пустых строк между смысловыми блоками), а нумерованные перечисления
«схлопнуты» или нумерация оторвана от содержимого пункта.

**⚠️ РЕЗУЛЬТАТ ДИАГНОСТИКИ:**

Парсер Obsidian (`app/ingest/parsers.py`) работает корректно:
- `[[wikilinks]]` разворачиваются в чистый текст (`[[REST|REST API]]` → `REST API`);
- YAML-frontmatter убирается;
- заголовки становятся метаданными раздела;
- структура списков и пустые строки между абзацами **сохраняются в чанках**.

Контекст, который получает LLM, отформатирован корректно. Модель также
генерирует ответ с корректной структурой (абзацы, нумерация).

**Корневая причина (подтверждена):** `filter_answer()` в
`app/generation/grounding.py`.

Прежняя реализация использовала flatten-подход:
1. `split_sentences()` разбивала ответ в **плоский** список фрагментов, разрезая
   И по переводам строк (`\n`), И по завершающей пунктуации (`.!?`);
2. затем `"\n".join(kept)` склеивал оставленные фрагменты одиночными `\n`.

Это уничтожало:
- **пустые строки между абзацами** (все схлопывались в один `\n`);
- **пункты списков с несколькими предложениями** (каждое предложение уходило
  на отдельную строку → визуально ломалась нумерация: «1.», затем содержимое
  на следующей строке, затем «2.» и т.д.).

### 6.2. ✅ Фикс `filter_answer` (код-уровень) — ВЫПОЛНЕНО
**Файл:** `app/generation/grounding.py`.

Новая реализация `filter_answer()`:
- **ходит по строкам** ответа (`answer.split("\n")`), сохраняя оригинальные
  переводы строк;
- **пустые строки** трактует как разделители абзацев и сохраняет их в выводе
  (как кандидаты — финальная очистка убирает дублирующиеся и краевые);
- **в пределах строки** разбивает содержимое на предложения (по `.!?`) и
  проверяет каждое на опору контекстом (`_sentence_supported`);
- **маркеры списков** (`«1.»`, `«-»`, `«*»`) приклеиваются к следующему за ними
  содержимому в пределах строки — нумерация не отрывается от текста пункта;
- неподдержанные предложения удаляются точечно, поддержанные остаются на
  своих местах (на той же строке);
- подряд идущие пустые разделители схлопываются в один, краевые обрезаются.

Статистика (`min_kept_ratio`, `min_kept_sentences`) считается по тем же
«предложениям» (units), что и раньше, — пороги семантики не изменились.

`split_sentences()` НЕ трогался — он остаётся для диагностики/логов и
определения коротких ответов (`< 3 предложений`).

### 6.3. ✅ Усиление промпта — ВЫПОЛНЕНО
**Файл:** `app/generation/prompt.py` (правило 6).

Правило 6 расширено явными инструкциями по форматированию:
- разделять смысловые блоки пустой строкой (абзац = одна мысль);
- перечисления оформлять списком (каждый пункт с новой строки, с маркером);
- не писать список в одну строку через запятую, если в контексте он задан

- между списком и следующим абзацем — пустая строка;
- не дублировать маркеры и не писать `\n` буквально как текст.

### 6.2.1. BUG: модель выдаёт перечисления одной строкой (инлайн)

**Симптом (из баг-репорта пользователя):**
```
JSON (JavaScript Object Notation) — это текстовый формат обмена данными...
В качестве значений в JSON могут быть использованы: - JSON-объект - Массив
- Число - Литералы... Well-formed JSON соблюдает следующие правила: 1. Данные
написаны в виде пар «ключ:значение»; 2. Данные разделены запятыми; 3. Объект
находится внутри фигурных скобок {}; 4. Массив — внутри квадратных [].
```

**Корневая причина:** модель `qwen2.5:3b-instruct` часто выдаёт перечисления
**инлайн** — одной строкой через маркеры (`- item - item` или `1. item 2. item`),
а не с переносами строк. Фильтр `filter_answer` (Этап 6) корректно **сохраняет**
существующие переносы, но переносов изначально нет в выводе модели. Промпт
(правило 6) даёт инструкции, но 3B-модель не всегда им следует.

**Решение:** детерминированный постпроцессинг — конвертация инлайн-списков
в многострочные на уровне кода, до скоринга/фильтрации.

### 6.2.2. ✅ `normalize_inline_lists()` — ВЫПОЛНЕНО
**Файл:** `app/generation/grounding.py`.

Новая функция `normalize_inline_lists(text)`:
- обрабатывает текст построчно;
- детектирует инлайн-списки по паттерну: `:` или `;` + пробел + маркер
  (`-`/`*` для маркированных, `N.` для нумерованных);
- разбивает инлайн-список, ставя каждый пункт на отдельную строку;
- первый маркер после `:` остаётся в начале списка, остальные ` - ` / ` N. `
  заменяются на `\n- ` / `\nN. `.

**Подключение:** в `app/generation/service.py`, функция `_clean()` —
нормализация применяется **до** `grounding_score` и `filter_answer`, чтобы
grounding видел корректную структуру ответа.

**Защита от ложных срабатываний:**
- дефисы в составе слов (`HTTP-сервер`, `клиент-серверный`) — не триггерят
  (инлайн-список ловится только при `:`/`;` + пробел перед маркером);
- числа без `:`/`;` перед ними (`Уровни: 0, 1, 2, 3. Версия 1.2`) — не
  считаются нумерованным списком;
- уже многострочные списки (`Принципы:\n- item\n- item`) — не дублируются
  (нет инлайн-паттерна `: -`).

## ЧАСТЬ 1.8. ЭТАП 7 — Остаточные нюансы форматирования (✅ ВЫПОЛНЕН)

> Живой тест (Этап 4.2) после Этапов 6/6.2 выявил 3 остаточные проблемы
> форматирования. Информация верная, но нюансы presentation quality мешают.

### 7.1. BUG: скачущая нумерация (маркер `N)` со скобкой)

**Симптом (из баг-репорта пользователя):**
```
REST (Representational State Transfer) - это парадигма...
1) Клиент-серверную архитектуру
2) Stateless (без состояния)
3) Кэширование
...
```

**Корневая причина (подтверждена анализом БД):** модель копирует стиль
источника дословно. Чанк 126 (`REST API.md`) хранит принципы как:
```
Принципы REST: 1) Клиент-серверная архитектура 2) Stateless 3) Кэширование ...
```
— **со скобкой, одной строкой**. `normalize_inline_lists()` (Этап 6.2)
распознавала **только маркер с точкой** (`N.`):
```python
_NUM_INLINE_RE = re.compile(r"([:;])[ \t]+(?=\d+\.[ \t]+\S)")  # ← нет \)
```
Маркер `1)` (со скобкой) **не распознавался** как старт инлайн-списка →
список не разбивался на строки → приходил сплошным текстом.

### 7.2. ✅ Фикс `normalize_inline_lists` (код-уровень) — ВЫПОЛНЕНО
**Файл:** `app/generation/grounding.py`.

- **`_NUM_INLINE_RE`** расширен: `\d+\.` → `\d+[.)]` (точка ИЛИ скобка).
- **`_split_inline_numbers`** обновлён: `re.sub(r"[ \t]+(\d+)[.)][ \t]+", ...)`
  ловит оба маркера и нормализует к `\n\1. ` (скобка → точка).
- **Нормализация `N)` → `N.`** (вариант А, одобрен пользователем): единообразный
  стиль списков во всех ответах бота, независимо от стиля источника. Промпт
  (правило 6) обновлён: «нумерованный, ВСЕГДА с точкой — не со скобкой `1)`».

### 7.3. BUG: дублирование логических выводов

**Симптом (из баг-репорта пользователя):**
```
...
Таким образом, основное отличие между уровнем 1 и уровнем 2 модели зрелости
REST заключается в использовании разных URL или HTTP-глаголов...

Таким образом, уровень 1 ограничивается использованием одного HTTP-глагола...
```

**Корневая причина:** `filter_answer()` проверяет каждое предложение на опору
контекстом, но **не проверяет на тавтологию между собой**. Оба «Таким
образом…» поддержаны контекстом по `word_coverage`, оба проходят фильтр — и
оба попадают в ответ, хотя говорят одно и то же.

### 7.4. ✅ Решение: промпт, не кодовый Jaccard-фильтр — ВЫПОЛНЕНО
**Файл:** `app/generation/prompt.py` (правило 7).

Первоначально была попытка решить кодовым Jaccard-фильтром
(`_dedupe_repeated_conclusions`), но она **убрана**: Jaccard на стемах не
отличает семантический повтор от разных фактов с общими словами и **рискует
вырезать полезные разные предложения** (например, «Уровень 1 использует POST»
и «Уровень 2 использует GET/POST/PUT/DELETE» — разные факты, высокий Jaccard).

**Итоговое решение:** промпт (правило 7) — явное указание модели
«НЕ ПОВТОРЯЙ одну и ту же мысль несколько раз; одна мысль — один абзац».
Для 3B-модели это безопаснее, чем агрессивный кодовый фильтр.

### 7.5. BUG: искажения терминов («Layered» → «Лякосящая»)

**Симптом:** модель иногда искажает слова при копировании из контекста
(«Layered System» → «Лякосящая система слоев»). Точечное искажение проходит
grounding, т.к. остальная часть предложения поддержана и покрытие высокое.

**Решение (промпт-уровень):** правило 8 «ДОСЛОВНОСТЬ терминов» — явное
указание копировать имена собственные, названия принципов, аббревиатуры и
формулировки из контекста **буквально**, не искажать слова и не выдумывать
расшифровки. Это превентивная мера; на уровне кода точечные искажения слов
выловить невозможно без полного словаря терминов.

### 7.6. ✅ Регрессионные тесты — ВЫПОЛНЕНО (2 теста)
**Файл:** `tests/test_anti_hallucination.py`.

- `test_normalize_inline_numbers_with_paren_marker`: `: 1) a 2) b` → разбивается.
- `test_normalize_paren_numbered_list_to_dot`: маркеры `1)` → `1.`.

Тесты на дедупликацию **не добавлялись** (фильтр убран — см. 7.4).

Все 79 тестов зелёные (77 + 2); `ruff` чист.

- ✅ `master_state.json` обновлён (Этап 7 в `completed_tasks`).
- ✅ `REFACTORING_PLAN.md` обновлён (этот раздел).
