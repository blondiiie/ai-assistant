## ЧАСТЬ 1. ИСТОРИЯ БАГА (✅ ИСПРАВЛЕН — Этап 5)

### 1.1. ✅ BUG: Заглушка на вопросах «что такое JSON» / «что такое HATEOAS»

**Симптом:** На вопрос «что такое JSON» бот выдаёт заглушку об отсутствии
информации, хотя заметка существует и проиндексирована.

**⚠️ РЕЗУЛЬТАТ ДИАГНОСТИКИ (2026-06-29, окружение поднято):**

| Вопрос | Retrieval | LLM raw | `has_foreign_script` | `grounding_score` | Итог |
|---|---|---|---|---|---|
| «что такое JSON» | ✅ найдены чанки 122/123/121 (`sim` выше 0.35) | ✅ корректный grounded-ответ | ❌ **True** (из-за `{}` и `[]`) | 0.984 | 🔴 **STUB** |
| «что такое HATEOAS» | ✅ найдены чанки 128/134 (`sim` выше 0.35) | ✅ корректный ответ | False | 0.671 | 🟢 **grounded** |

**Корневая причина (подтверждена):**

1. **JSON — виновник `has_foreign_script`.** В `app/generation/grounding.py`
   жёсткий regex `_ALLOWED_CHARS_RE` разрешал только узкий набор пунктуации
   и **не включал** `{`, `}`, `<`, `>`, `=`, `*`, `+`, `|`, `@`, `#`, `$`,
   `%`, `^`, `~`, `` ` ``. Эти символы легитимны в технических заметках
   (примеры JSON, YAML, HTML, HTTP, шаблоны) и модель правомерно воспроизводит
   их в ответе. Гейт создавался для ловли утечки иероглифов/арабицы, но
   ошибочно срубал любые ответы с технической пунктуацией. Все 3 попытки
   генерации JSON-ответа блокировались именно этим гейтом, хотя
   `grounding_score=0.984` (word_coverage=1.0, trigram=0.95, distinctive=1.0).

2. **HATEOAS — баг НЕ воспроизводился.** Уже был починен на Этапе 2
   (единый `grounding_score` + короткие ответы < 3 предложений без
   `min_kept_sentences`). Регрессионный тест добавлен для защиты (см. 5.4).

**Исключённые гипотезы (диагностикой опровергнуты):**
- ❌ Retrieval не находит чанк — опровергнуто: JSON-чанки 121/122/123 и
  HATEOAS-чанки 128/134 находятся, `sim` выше `sim_threshold=0.35`.
- ❌ Чанкование разрезает определение — опровергнуто: определения JSON и
  HATEOAS лежат в БД целиком (id 122 и 128).
- ❌ `grounding_threshold=0.5` отсекает валидный ответ — опровергнуто: score
  0.984 (JSON) и 0.671 (HATEOAS) оба выше порога.

### 5.2. ✅ Фикс `has_foreign_script` (код-уровень) — ВЫПОЛНЕНО
**Файл:** `app/generation/grounding.py`.

### 5.3. ✅ Усиление retrieval для коротких терминов — ВЫПОЛНЕНО (превентивно)
**Файл:** `app/retrieval/service.py`.

- **Keyword-boost по `content`** (не только по `source_name`/`section`): точное
  вхождение термина в текст чанка даёт буст гибридному скору (до +0.15).
  Лечит слабый эмбеддинг короткого термина, где стоп-слова «что»/«такое»
  доминируют в векторе вопроса.
- **`plainto_tsquery('simple', ...)` для латинских терминов**: английские
  «JSON»/«HATEOAS» плохо стеммятся russian-конфигом → low `lex` → слабый
  гибридный скор. Добавлен как второй аргумент к `plainto_tsquery('russian')`
  в `CANDIDATE_SQL` и `NEIGHBOR_CHUNKS_SQL`.

> Обоснование обязательности: даже если на текущих 22 документах retrieval
> находит чанки, на более богатом корпусе (и при иных формулировках) короткий
> термин с доминирующими стоп-словами — классическая точка отказа RAG.

### 4.2. ⏳ Живой тест в Telegram — РУЧНАЯ ПРОВЕРКА

### 4.3. ✅ Профиль RAM — ИНСТРУМЕНТ ГОТОВ
Создан `scripts/profile_ram.py` (psutil) + Makefile-цель `profile-ram`.

**Решение проблемы «зацикливания»** (почему предыдущие проверки RAM не
заканчивались): предыдущий подход требовал ручного наблюдения за Activity
Monitor без чётких критериев завершения. Новый скрипт автономен:
- жёсткий лимит итераций (`--requests`, дефолт 5);
- жёсткий таймаут на каждый запрос `asyncio.wait_for` (`--per-request-timeout`);
- жёсткий общий таймаут через SIGALRM-watchdog (`--total-timeout`);
- детерминированный exit-code (0 — нет утечек и пик в зелёной зоне);
- отчёт в `data/ram_profile.json`.

Зелёная зона Python-RSS: ≤ 400 МБ (веса модели в Ollama — отдельный процесс ~2 ГБ).
Порог утечки: монотонный рост RSS > 10 МБ с нестабилизирующимся хвостом.

Запуск:
```bash
make profile-ram
# или с параметрами:
uv run python -m scripts.profile_ram --requests 10 --per-request-timeout 180
```

> Критерий успеха: ответы совпадают с эталоном по смыслу; RAM в зелёной зоне;
> 3 повтора после рестарта дают одинаковый (на попытке 0) результат.
